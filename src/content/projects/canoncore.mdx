## What it is

CanonCore is what you'd get if Plex and Obsidian had a child. The Plex half is a file-backed personal media library. It pulls metadata from pluggable providers (TMDB for film and TV, Big Finish for audio drama), plays video and audio, tracks what I've watched, and scrobbles. Unlike Plex, it doesn't need a home server or a NAS. Google Drive is the storage backend; new files in the connected folder appear within seconds via webhook sync.

The Obsidian half is the knowledge graph. Eight kinds of **entity** are first-class rows: person, character, franchise, company, concept, location, era, and genre. **Relationships** between items come in nine typed flavours: sequel, prequel, spin-off, crossover, remake, adaptation, behind-the-scenes, version, and deleted scenes. The library renders as a force-directed graph; the data model is a DAG, not a tree, so the same film can be **multi-placed** in multiple containers without being duplicated. People, characters, and franchises aren't containers; they're entities, browsed sideways through the entity index.

Two apps share the model: a web app and a native mobile app. Both render the same library and the same graph view. I built it for myself.

## Why it was hard

Three constraints set the architecture.

**Multi-placement.** A media library is rarely a tree. An episode of anthology TV lives in a season, a series, "2003 releases", and a "Best of the year" shortlist all at once. A tree forces duplicates or abandons the second dimension. The data model is a DAG with an edge table; cycle detection, DAG-aware visibility, and reparenting UX come with the shape.

**Cross-platform, without two codebases.** I wanted the same library on web and mobile without writing it twice. Data and business logic are shared packages; UI is not. A native-feeling app wants native primitives, not a lowest-common-denominator wrapper. The graph view is the lone exception (covered below), mounted on mobile through Expo's `"use dom"` directive.

**Metadata from many places.** TMDB covers film and TV but nothing else. Hardcoding a new integration for every other media type would have wedged provider quirks deep into the enrichment code. So I defined a single port, `CMPPProvider`, and let adapters plug in.

## Architecture decisions

### Metadata as ports and adapters

`CMPPProvider` is the interface for metadata sources. Two adapters today: TMDB (film and TV) and Big Finish (audio drama). 

TMDB wraps upstream calls in a 5-failure / 60-second circuit breaker, and external IDs are deterministic (`tv:{showId}:season:{n}:episode:{n}`) so re-enrichment hits the same entity without a second search. 

Big Finish was the real test. No public API; the data lives behind an open-source aggregator that exposed products but not the series-level groupings (Big Finish "ranges") a real library browses by. I forked it, added `/bigfinish/range/{slug}` and `/bigfinish/ranges`, and the adapter sits on top. None of that leaks past the boundary, and planned Audible, Open Library, and Google Books adapters can sit on the same fork: one fork, three more providers, no new fetch layer. 

Enrichment runs through an Inngest pipeline with four steps (extract colour for theming, auto-populate entities, check for nested children, fire a nested-scaffold pipeline if so), each retrying independently so a flaky TMDB call doesn't wipe out colour extraction that already succeeded.

### DAG with recursive-CTE visibility

The library is a DAG. Items connect through an `itemEdge` adjacency table; a `(parentId, sortOrder)` index keeps ordered sibling reads (the most common query) cheap. The same item can appear under multiple parents (multi-placement). Cycle detection runs at mutation time through a recursive CTE over ancestors.

The more interesting CTE is visibility: it walks ancestor paths through *public edges only*. The item, the owner, and at least one full ancestor path all need to be public, so an item placed in both a public collection and a private one stays visible from the public side. The batch form does it for N items in one query, which keeps the public Explore feed O(N) instead of O(N × depth).

Typed relationships between items (sequel, prequel, etc.) live in a separate table; the two directions of any pair round-trip through one record via an inverse-type lookup.

### Drive sync, event-driven and crash-safe

Google Drive sits behind a `StorageProvider` interface, so application code never imports the SDK directly.

Webhooks from Drive map to a connection into Inngest with concurrency limited to one per connection. Changes are deduplicated (last write wins per fileId) and sorted folders-first, because a file arriving before its parent folder would hit a foreign-key violation. Push notification channels auto-renew every six hours via Inngest cron.

Duplicate detection runs *after* the main sync transaction commits, so a crash mid-sync just means the duplicate is re-detected on the next run.

Conflicts (same file edited on both sides) surface in a side-by-side diff UI with three strategies (Keep Drive, Keep Canon, Keep Both); the user picks rather than the sync guessing.

Bidirectional push uses the same interface. After a push, `driveFileName` is updated to match, so the resulting webhook ping looks like a no-op and skips.

### Colour-adaptive theming

Every enriched item has a backdrop, and the UI tints to match: buttons, borders, ambient glow. On the server, `sharp.stats()` extracts a dominant colour from the artwork once at enrichment time (with a saturation boost so muted posters don't collapse to grey), and the client expands that one hex into 10 shades from light to dark. Those shades drive CSS custom properties registered through `@property`, which makes them animatable; navigating between items crossfades the theme over 500ms. OG images use the same shades, so shared links match.

### Entities with retry-safe enrichment

Entities are rows in an `entity` table (eight types, listed earlier). Items link to entities through an `itemEntity` join table with eight roles (STARS_AS, APPEARS_IN, DIRECTED, WROTE, COMPOSED, PRODUCED_BY, SET_IN, FEATURES). A TMDB match auto-populates `PERSON`s for the cast, a `CHARACTER` per role, and `PERSON`s for the crew; companies, franchises, locations, and the rest are user-created. Both dedup by `(userId, cmppProvider, cmppExternalId)` where a provider ID exists, by `(userId, name, type)` otherwise. Cast is capped at twenty per item, because TMDB will cheerfully return forty and most of them are "Uncredited Barista".

On re-enrichment (the user picked a different TMDB match), stale entity role rows are deleted, but only for provider-sourced roles, the ones a re-enrichment is allowed to churn. Manually-added entity roles survive, and typed item relationships (sequel, adaptation, and the rest) are left alone entirely. The nested-scaffold pipeline itself is retry-safe: if a step fails mid-way and Inngest retries, it picks up from the failed step instead of duplicating seasons.

### Graph view

Force-directed graph, PIXI.js v8 + d3-force. Nodes are items and entities; edges fall into three classes (containment, typed relationships, item→entity). The simulation is BFS depth-limited from a focal node, so a library of thousands of items doesn't collapse into a hairball.

Mobile would normally mean a second WebGL port through Skia. Instead the component is wrapped in an Expo DOM component (`"use dom"`) and the same renderer runs inside a native WebView.

### Cross-platform data contract

Shared packages, consumed by both apps:

- `@canoncore/api`: tRPC routers, 12 of them, 96 procedures.
- `@canoncore/db`: Drizzle schema, 17 tables, 8 enums, 20 migrations.
- `@canoncore/hooks`: shared TanStack Query hooks and colour math. No DOM APIs.
- `@canoncore/store`: Zustand + Immer stores, platform-agnostic.
- `@canoncore/validators`: Zod schemas (auth, items, entities, relationships, profiles).
- `@canoncore/graph-engine`: the PIXI renderer, framework-agnostic.
- `@canoncore/services`: server-only. Storage, CMPP, sync, visibility, colour, crypto.
- `@canoncore/inngest`: event schemas and durable functions.

Platform boundaries are marked with `server-only` imports. Mobile can't accidentally import the Drive provider or the Drizzle client, because the bundler errors first.


## Auth and security

[Better Auth](https://www.better-auth.com/) with email/password and the Expo plugin. Argon2id password hashing (the library default). Password reset has a 30-minute token expiry and revokes all active sessions on success (`revokeSessionsOnPasswordReset: true`); if the user reset because they suspected compromise, old sessions can't survive the reset. Account lockout is five failures in fifteen minutes.

Some fields (visibility, bio, avatars, lockout counters) should only be written by the server. The Better Auth config blocks client writes, and a hook strips them from update bodies as backup. Standard defence against mass assignment.

**Drive OAuth tokens** are encrypted at rest with AES-256-GCM (authenticated, so tampering is detected). Each token gets a fresh IV; the key comes from the environment.

**Rate limiting** uses [Arcjet](https://docs.arcjet.com/reference/nextjs/) (120 req/min per IP) at the tRPC handler, with Better Auth adding a stricter limit on auth endpoints. 

**Input validation** is Zod on every API procedure; no `z.any()` or `z.unknown()` in the router. Public-page JSON-LD escapes `</` to `<\/` so user bios can't break out of the script tag.

## Media playback

Web uses [Vidstack](https://www.vidstack.io/) whilst mobile uses [expo-video](https://docs.expo.dev/versions/latest/sdk/video/) for video and [react-native-track-player](https://rntp.dev/) for audio, coordinated through a bridge that shares a Zustand store with the web app (queue, currentIndex, shuffleMode, repeatMode, isPlaying, isExpanded, isQueueOpen). The store has to outlive the player component, because the mini-player, expanded view, and queue sheet all read from it.

A few specifics:

- **Scrobble at 80% completion**, once per item (the Last.fm/Trakt convention, so it lands where users expect). The Inngest function is keyed by user+item, so fast scrubbing can't double-count.
- **Position persistence every 30 seconds**, plus an immediate save on visibility-change (browser) and `AppState.change` to background (mobile). The immediate save covers force-quit.
- **Subtitles**. Web uses Vidstack `<Track>` elements; mobile parses VTT/SRT with `@plussub/srt-vtt-parser` and renders an overlay, because expo-video doesn't support external subtitle URLs.
- **Range-request streaming**. `/api/media/[fileId]` forwards `Range` headers straight to Drive's `files.get(alt=media)` and pipes the response through without buffering. `Cache-Control: private, max-age=3600` stops the browser from re-fetching. Permission is checked per request through one indexed `itemFile → item → userId` join, so a leaked URL doesn't grant anonymous access; the recipient still has to be the owner.

## State, forms, URLs

- **Server state**: TanStack Query through tRPC, 30-second `staleTime`, SuperJSON hydration, streaming link through `httpBatchStreamLink`.
- **Client state**: Zustand + Immer for the playback queue, shared between web and mobile.
- **Form state**: TanStack Form with Zod resolvers.
- **URL state**: `nuqs` for filter, sort, and view-mode params that should survive a refresh.

Optimistic mutations make the UI feel instant: the cache updates before the server replies, and on failure the toast tells you it didn't stick. Seventeen flows across the two apps (create, delete, reparent, reorder, watch, visibility), each wrapping `onMutate` / `onError` / `onSettled` with toast on rollback ([sonner](https://sonner.emilkowal.ski/) on web, sonner-native on mobile).

## Performance

The mini-player progress bar updates every frame. It bypasses React state and writes to the DOM directly via a Vidstack subscription, so the transport controls around it don't re-render sixty times a second.

## Accessibility

Full keyboard support in the player, focus management around dialogs, skip links, visible focus rings, and a shortcut handler that defers to Vidstack when the player is focused. Every animation (colour crossfade, Ken Burns hero, stagger reveals) respects `prefers-reduced-motion`. The graph view carries a dismissible notice pointing screen-reader users to the item-detail "Related" section, which exposes the same data as a list.

Mobile has `accessibilityLabel` and `accessibilityRole` on interactive elements throughout.

## Testing

The shape matters more than the total:

| Layer | Files | Cases | Runner | Database |
|---|---|---|---|---|
| Integration (tRPC) | 18 | 326 | Vitest | PGlite, in-process |
| Unit | 21 | 252 | Vitest | n/a |
| Inngest steps | 7 | 29 | `@inngest/test` `InngestTestEngine` | Step mocks |
| Playwright E2E | 66 | 392 | Playwright | Postgres container per shard (CI) |
| Mobile E2E | 45 flows | n/a | Maestro | Neon branch |

Integration tests outnumber unit tests deliberately. They run tRPC procedures against in-process Postgres ([PGlite](https://electric-sql.com/docs/product/pglite)) instead of mocks, catching bugs across procedure logic, Drizzle, and SQL in one layer. No Docker on the dev path.

Inngest functions are tested step-by-step with `InngestTestEngine`: mocked service calls, verified retries and error paths, no dev server. Most projects that run Inngest skip this entirely.

Playwright E2E is split `@ci` (chromium, per-PR) and `@nightly` (external services). CI runs the `@ci` set across a 4-shard matrix with per-shard Postgres containers (covered below). Shards merge into one HTML report at the end.

Maestro covers mobile through reusable sub-flows (fourteen of them) so auth and setup aren't duplicated across thirty-one top-level flows.

## Observability

- **Sentry**: errors and traces, 10% sample in production, 100% in dev, fully disabled in E2E. Session replays at 10% always, 100% on error.
- **LogTape → Better Stack**: structured logs through an OpenTelemetry sink, category-prefixed, level-gated per environment.
- **Inngest**: its own function-run observability. Step traces, retries, and dead-letter queues all visible from the dashboard.

## CI/CD

GitHub Actions with jobs for lint, typecheck, build, unit and integration tests, knip (unused exports), commitlint, and the 4-shard E2E matrix (using the official `mcr.microsoft.com/playwright` container). Pre-commit hooks ([Lefthook](https://github.com/evilmartians/lefthook)) run a lighter version of the same gates locally.

Deploy is [Vercel](https://vercel.com/) for the web app.

## Neon branch per environment

Four database environments, each a branch of the same schema (so demo data doesn't pollute development and E2E doesn't race against my own usage):

- `main`: production
- `development`: default for local dev
- `demo`: seeded showcase data
- `e2e`: Playwright and Maestro target this locally; integration tests use PGlite in-process instead.

## What I'd do differently

- **Model the DAG from the first migration.** v1 was a tree. v2 was the rebuild. Knowing what I know now, the edge table goes in on day one and the rework epic doesn't happen.
- **Storybook before the components, not after.** Deferring it meant every component got tested in-app first. The ones that landed cleanly are the ones I'd have built in isolation anyway.
